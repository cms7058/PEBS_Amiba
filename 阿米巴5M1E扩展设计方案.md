# 阿米巴动态智能体系统 — 5M1E 扩展设计方案

> 副标题：从「组织变革咨询」升级为「基于成本控制的人机料法环阿米巴管理职能体」
> 版本：V2.0（在 [V1.0 设计方案](阿米巴动态智能体系统设计方案.md) 基础上扩展）
> 日期：2026年6月
> 适用行业：汽车零部件制造业 × 项目制非标设备制造业

---

## 〇、本次扩展要解决的问题

V1.0 系统本质是**以组织变革为中心的咨询诊断工具**：六维诊断（组织/财务/信息化/设备/流程/文化）→ 四层引擎（诊断→规划→设计→部署）→ 输出方案与陪跑。

它的两个结构性短板：

1. **诊断停留在"管理层访谈"层，没有触达车间现场的真实数据。** 隐性成本、岗位冗余、工作流效率、排料损耗这些痛点，靠问卷问不出来，必须有现场数据采集与分析工具支撑。
2. **四个已开发的工具（APS/BOM/Lean/Worktime）各自为战，没有汇入阿米巴的成本核算骨架。** 它们恰好覆盖了制造业 5M1E 的核心采集与执行能力，但目前与阿米巴没有数据契约。

本次扩展的目标：

> **把诊断维度从"6 个管理维度"重构为"5M1E 现场要素 × 成本控制主线"，并把四个已有工具作为子智能体编排进阿米巴框架，让阿米巴第一次能基于车间真实数据做成本归因、冗余诊断和内部核算。**

## 一、核心框架：5M1E × 三性 × 成本控制

### 1.1 5M1E（六要素）与现有六维的映射关系

V1.0 的六维是**企业管理视角**，新增的 5M1E 是**现场作业视角**，两者不是替换而是**正交叠加**——管理维度回答"组织能不能切阿米巴"，现场要素回答"切出来的阿米巴成本算不算得清"。

| 5M1E 要素 | 中文 | 现场诊断焦点 | 主力子智能体 | 关联 V1.0 维度 |
|---|---|---|---|---|
| **Man** | 人 | 工时基准、岗位冗余、人员冗余、技能-工序匹配、人均产值 | **Worktime（视频工时）** | 组织 / 文化 |
| **Machine** | 机 | OEE、设备共用归属、设备内部租赁价、能耗归因、产能瓶颈 | **APS（排产）** | 设备 |
| **Material** | 料 | 排料/下料利用率、库存周转、余料管理、BOM 准确性、辅料定额 | **BOM + APS** | 财务（隐性成本） |
| **Method** | 法 | 工作流节点合理性、IO 交付物三性、工序标准化、供应链流程三性 | **Lean（精益方法论）** | 流程 |
| **Environment** | 环 | 能源计量、现场 5S/安全、环保合规对成本的影响 | **Lean + APS（能耗）** | 设备 / 流程 |
| **Measurement** | 测 | 数据采集成熟度、计量器具、核算数据质量（成本控制的前提） | 贯穿所有工具 | 信息化 / 财务 |

> 说明：经典「人机料法环」是 4M1E（5要素）。本方案显式补入 **Measurement（测量）** 成为 5M1E，因为阿米巴成本核算的成败 100% 取决于"测得准不准"——它正好对应 V1.0 的信息化成熟度，是成本控制的地基。

### 1.2 「三性」诊断范式（贯穿法、料、供应链）

你提出的"节点三性判断"是本次方法论的核心工具，统一定义为：

| 三性 | 判定问题 | 输出 |
|---|---|---|
| **合理性** Rationality | 这个节点/岗位/工序该不该存在？设置位置对不对？ | 冗余清单、合并建议、价值流定位（VA/NVA/NNVA） |
| **完整性** Completeness | 该有的节点/输入/输出/管控点缺没缺？ | 缺口清单、断点风险、缺失管控点 |
| **正确性** Correctness | 节点的 IO 交付物、参数、责任人对不对？ | 错误/不一致清单、返工根因 |

三性判断的对象贯穿三个层面：

- **工作流层（法）**：每个流程节点、节点的输入/输出交付物、节点责任岗位
- **物料流层（料）**：排料方案、下料指令、库存策略、余料处置
- **供应链层**：从供应商到客户的端到端节点（寻源→采购→入厂→生产→交付）

### 1.3 成本控制主线（系统的脊柱）

所有诊断与工具最终都要回答一个问题：**这一项要素的浪费，折算成多少钱，归到哪个阿米巴？** 因此引入统一的**成本归因主线**贯穿全系统：

```
现场要素浪费 ──量化──► 成本科目 ──归因规则──► 责任阿米巴 ──► 内部核算/分红
   ↑                                                          │
   └──────────── 子智能体提供真实数据 ◄──── 改善闭环 ◄────────┘
```

每个 5M1E 要素都挂一组**成本科目**与**归因规则**（详见第四章）。这把 V1.0 第六章"隐性成本显性化"从专题升级为**系统主干**。

### 2.1 诊断结构：管理层 + 现场层双轨

### 2.2 「法」维度扩展：工作流 + 岗位 + 供应链三性诊断

这是你重点提出的方向，单独展开。新增三个诊断子模块（主力工具：Lean，数据辅以 APS/Worktime）：

#### ① 工作流节点诊断
对企业现有主流程（订单→交付、采购→入库、设计→出图等）做结构化建模，逐节点判三性：

| 诊断项 | 方法 | 输出 |
|---|---|---|
| 节点设置合理性 | VSM 价值流分析，标注每节点 VA/NVA/NNVA | 可删除/可合并节点清单 |
| 节点 IO 交付物完整性 | 检查每节点输入是否齐备、输出是否被下游真正使用 | 缺失输入清单、僵尸输出清单（产出但无人用） |
| 节点交付物正确性 | 交付物模板/字段/参数符合度抽查 | 错误模板清单、返工根因 |
| 工作流效率 | 节点 L/T、等待时间、流动效率（增值时间÷总周期） | 流动效率%、瓶颈节点、节拍失衡点 |

#### ② 岗位/人员冗余诊断
把"组织维度"从访谈层下沉到"岗位-工序-工时"实测层：

| 诊断项 | 方法 | 数据来源 | 输出 |
|---|---|---|---|
| 岗位冗余 | 岗位职责映射到流程节点，找无价值节点对应岗位 | 工作流诊断 + 组织架构 | 应撤/应合并岗位清单 |
| 人员冗余 | 实测工时负荷率 = 有效作业工时 ÷ 出勤工时 | **Worktime 视频工时** | 各岗位负荷率、富余人力数 |
| 技能-工序错配 | 技能矩阵 vs 工序要求 | 组织 + APS 工艺 | 错配点、培训/调岗建议 |
| 人均产值 | 阿米巴产出 ÷ 在岗人数 | 财务 + 组织 | 人效 benchmark 对比 |

> 工时负荷率是岗位/人员冗余诊断的关键证据，这正是 **Worktime 项目**的核心价值——用固定机位视频自动切分工步、统计循环工时，无需上 MES 即可拿到现场实测工时。

#### ③ 供应链流程三性诊断
端到端节点（寻源→比价→采购→到货→检验→入库→领用→生产→发运→回款）逐节点判三性，重点：

- **合理性**：采购集中/分散是否合理、是否存在不增值审批层
- **完整性**：是否缺到货检验、缺安全库存策略、缺供应商绩效节点
- **正确性**：采购参数（MOQ、提前期、价格）是否准确、是否导致呆滞或断料

### 2.3 「料」维度扩展：排料/下料/库存/余料

主力工具：**BOM（物料结构与用量）+ APS（排料排产与库存联动）**

| 子领域 | 诊断/优化项 | 工具 | 成本科目 |
|---|---|---|---|
| 排料 Nesting | 板材/型材套料利用率、共边/共线优化 | APS 排料 + BOM | 材料利用率损失 |
| 下料 Cutting | 下料指令与 BOM 一致性、批量合并下料 | BOM + APS | 多余下料、错料返工 |
| 库存 Inventory | 周转率、呆滞料、安全库存合理性、ABC 分类 | APS + ERP 接口 | 库存资金占用、呆滞计提 |
| 余料 Remnant | 余料登记/编码/再利用率、余料库 | BOM（余料 BOM）+ APS | 余料浪费、重复采购 |
| 辅料/耗材 | 定额消耗 vs 实际、超标预警 | 沿用 V1.0 定额/条码法 | 辅料隐性成本 |

> 余料管理是制造业普遍的隐性浪费点：BOM 项目可建立"余料 BOM"对余料编码登记，APS 排料时优先调用余料库，闭环掉"切了余料→堆在角落→下次重新采购"的损失。

### 2.4 「人/机/环/测」其余要素的诊断要点

| 要素 | 关键诊断项 | 工具 | 成本科目 |
|---|---|---|---|
| 人 Man | 工时负荷率、工步增值比、PMTS 标准工时 vs 实测 | Worktime | 人工浪费、加班成本 |
| 机 Machine | OEE（可用率×性能×良率）、设备共用归属、产能瓶颈、换线时间 | APS | 设备折旧分摊、产能损失 |
| 环 Environment | 三级能源计量缺口、5S/安全隐患、环保合规成本 | APS（能耗）+ Lean | 能源成本、合规成本 |
| 测 Measurement | 数据采集成熟度（L1/L2/L3）、计量器具、核算数据质量 | 贯穿 | —（决定其它成本算得准不准） |

## 三、四个子智能体的编排集成（架构）

采用**子智能体/模块编排**方式：四个项目保持独立部署与独立技术栈，amiba 作为**编排层 + 成本核算中台**，通过 API/数据契约调用它们，结果回流到阿米巴。

### 3.1 总体架构

```
┌──────────────────────────────────────────────────────────────┐
│                    阿米巴编排层 (PEBS_amiba)                    │
│  诊断引擎V2 │ 规划引擎 │ 设计引擎 │ 部署引擎 │ 成本核算中台    │
└───────┬───────────┬───────────┬───────────┬──────────────────┘
        │ 数据契约   │           │           │
   ┌────▼────┐ ┌────▼────┐ ┌────▼────┐ ┌────▼─────┐
   │Worktime │ │  APS    │ │  BOM    │ │  Lean    │
   │ 人/工时 │ │ 机/料/环│ │  料     │ │ 法/环    │
   │(FastAPI)│ │(agent-  │ │(FastAPI)│ │(Express) │
   │         │ │ server) │ │         │ │          │
   └─────────┘ └─────────┘ └─────────┘ └──────────┘
        │           │           │           │
        └───────────┴─────┬─────┴───────────┘
```

### 3.2 每个子智能体的集成契约

| 子智能体 | 现有技术栈 | 向阿米巴提供（输入契约） | 阿米巴消费场景 |
|---|---|---|---|
| **Worktime** | Python/FastAPI :8000 | 工步切分、循环工时、工时负荷率、PMTS 对比、异常 | 人维诊断、岗位/人员冗余、人工成本核算 |
| **APS** | Node agent-server :8787 | OEE、产能瓶颈、排料利用率、库存周转、能耗（如接入） | 机/料/环诊断、设备内部租赁价、产能成本 |
| **BOM** | FastAPI :8000 + Next 前端 | BOM 结构、标准用量、余料 BOM、辅料定额 | 料维诊断、材料成本核算、余料闭环 |
| **Lean** | Express + React :5173 | 流程三性诊断、VSM、鱼骨/Pareto 根因、8D/DMAIC 报告、RAG 知识库 | 法/环诊断、改善方案、知识沉淀 |

### 3.3 数据契约设计（统一中间模型）

阿米巴侧定义一组**标准中间数据模型**（与现有 `diagnosis-types.ts` 同层，新增 `factory-types.ts`），各工具的输出适配到这套模型，避免阿米巴耦合各工具内部结构：

```ts
// 统一现场要素数据模型（建议新增 app/src/lib/factory-types.ts）
type Factor = "man" | "machine" | "material" | "method" | "environment" | "measurement";

interface FactorMetric {
  factor: Factor;
  key: string;              // 如 "labor_load_rate" / "oee" / "nesting_yield"
  label: string;
  value: number;
  unit: string;
  benchmark?: number;       // 行业基准
  source: "worktime" | "aps" | "bom" | "lean" | "manual";
  amibaId?: string;         // 归属阿米巴
  capturedAt: string;
}

interface WasteItem {            // 浪费项 → 成本归因主线的载体
  factor: Factor;
  three_props: "rationality" | "completeness" | "correctness";
  description: string;
  annualCost: number;          // 年化成本（元）
  costAccount: string;         // 成本科目
  attributionRule: string;     // 归因规则
  responsibleAmibaId?: string;
  improvementRef?: string;     // 关联 Lean 改善方案/8D 报告
}
```

> 集成可分阶段落地：先做**手工/文件导入**（各工具导出 JSON/CSV，阿米巴导入）跑通数据契约，再升级为 **API 直连**，最后做**实时/定时同步**。这与 V1.0 的 L1/L2/L3 数据采集分级一致。

## 四、成本控制主线（系统脊柱）

### 4.1 5M1E 成本科目地图

| 要素 | 成本科目 | 数据来源 | 默认归因规则 |
|---|---|---|---|
| 人 | 直接人工、加班、闲置人工 | Worktime 负荷率 | 按阿米巴实测有效工时 |
| 机 | 设备折旧、维修、产能损失 | APS OEE | 设备内部租赁制（V1.0 §6） |
| 料 | 直接材料、利用率损失、呆滞、余料浪费、辅料 | BOM+APS | 排料利用率差额归制造阿米巴；呆滞按责任归采购/计划 |
| 法 | 返工、协调损耗、僵尸交付物工时 | Lean+Worktime | 返工归责任工序；协调按会议成本法 |
| 环 | 能源、合规、5S 损失 | APS 能耗 | 三级计量分摊（V1.0 §6） |
| 测 | 数据采集改造投入 | 信息化诊断 | 资本性投入，按改造档次 |

### 4.2 成本归因引擎工作流

```
1. 子智能体上报 FactorMetric（实测值 + 基准）
2. 引擎按规则把"实测-基准"差额折算成 WasteItem.annualCost
3. 按 attributionRule 归到 responsibleAmibaId
4. 汇入阿米巴内部核算（转让价/分摊/分红）
5. Lean 生成改善方案 → 闭环追踪 → 复测 FactorMetric → 验证改善金额
```

### 4.3 与内部转让定价的衔接

V1.0 已定义内部转让价 = 标准工时成本 ×(1+内部利润率)。扩展后：
- **标准工时成本**由 Worktime 的 PMTS 标准工时校准（不再拍脑袋）
- **设备工时单价**由 APS 的 OEE/可用工时校准（设备内部租赁制落地）
- **材料成本**由 BOM 标准用量 + APS 排料利用率校准
- 三者使内部定价第一次有了**现场实测依据**，而非纯财务估算

## 五、四层引擎的相应升级

| 引擎 | V1.0 | V2.0 扩展 |
|---|---|---|

## 九、附录：插拔式对接架构（独立可用 + 接入即上传）

本章回答一个核心问题：**四个工具如何做到「不接阿米巴时是完整独立产品，接上阿米巴时自动把数据上传给阿米巴处理」？**

### 9.1 设计原则：四个工具对阿米巴「零硬依赖」

插拔式的本质是**依赖方向单向、且可选**：

```
独立模式：  [工具] ──产出──► 自己的界面/导出文件        （阿米巴不存在也完全正常）
接入模式：  [工具] ──产出──► 自己的界面 + 推送阿米巴      （多一个旁路出口而已）
```

四条硬规则：
1. **工具不 import 阿米巴的任何代码**，只依赖一份独立的"阿米巴连接器 SDK"（薄客户端）。
2. **连接器靠环境变量激活**：没配 `AMIBA_ENDPOINT` 就完全休眠，工具行为与现在一模一样。
3. **上传是旁路、非阻塞**：推送失败只记日志、进本地重试队列，绝不影响工具自身功能。
4. **契约稳定**：工具只认 §3.3 的中间模型（FactorMetric/WasteItem），不关心阿米巴内部如何存储/核算。

> 这样四个工具仍可各自独立销售/部署/演进；阿米巴只是它们的一个"可选数据下游"。

### 9.2 连接器开关（配置驱动的插拔点）

每个工具新增同一组环境变量（命名统一）：

```bash
# 不配 = 独立模式；配上 = 接入阿米巴模式
AMIBA_ENDPOINT=https://amiba.example.com   # 阿米巴中台地址
AMIBA_TOKEN=amk_xxx                         # 接入令牌（阿米巴后台生成）
AMIBA_ENTERPRISE_ID=ent_xxx                 # 数据归属哪家企业
AMIBA_SOURCE=worktime                       # 本工具标识（worktime/aps/bom/lean）
AMIBA_SYNC_MODE=push                        # push(实时推) | manual(只导出文件) | off
```

工具启动时探测：`AMIBA_ENDPOINT` 为空 → 连接器休眠；非空 → 连接器激活、启动时向阿米巴 `/api/connectors/hello` 注册一次（能力发现，见 9.6）。

### 9.3 三种对接档次（与 V1.0 数据采集分级一致）

| 档次 | 机制 | 工具侧改动 | 适用 |
|---|---|---|---|
| **L1 文件互通** | 工具导出标准 JSON/CSV，阿米巴页面手工导入 | 仅加一个"导出为阿米巴格式"按钮 | 内网隔离、POC、最快跑通契约 |
| **L2 主动推送 Push** | 工具产出结果后，连接器 POST 到阿米巴 ingest API | 加连接器 SDK，加环境变量 | 大多数场景的默认 |
| **L3 拉取/订阅 Pull** | 阿米巴侧适配器定时调用工具自身 API 拉数 | 工具几乎无改动（暴露只读 API 即可） | 工具不便外连、阿米巴主导编排 |

三档可共存：先用 L1 验证数据格式，再切 L2 自动化，对不便外连的工具用 L3。

### 9.4 阿米巴侧：统一接入 API（ingest）

阿米巴新增一组连接器 API（与现有 `app/src/app/api/` 风格一致，`runtime="nodejs"`），用**连接器令牌**鉴权（区别于用户登录 session）：

```
POST /api/ingest          # 接收一批 FactorMetric / WasteItem（L2 push 主入口）
POST /api/connectors/hello# 工具启动注册：上报 source、版本、能力清单
GET  /api/connectors      # 管理端查看：哪些工具已接入、最近上报时间、健康度
POST /api/ingest/file     # L1 文件导入（页面上传 JSON/CSV → 复用同一校验/落库逻辑）
```

`POST /api/ingest` 的请求信封：

```jsonc
{
  "source": "worktime",
  "enterpriseId": "ent_xxx",
  "batchId": "wt_20260614_001",   // 幂等键：同 batchId 重复上传只更新不重复
  "schemaVersion": "v2",
  "metrics":  [ /* FactorMetric[] */ ],
  "wasteItems": [ /* WasteItem[] */ ]
}
```

落库（沿用现有文件存储 `storage.ts` 的 `atomicWriteJSON`，无需引数据库）：
- `data/factory/<enterpriseId>/metrics.json`、`waste.json`（按 source + batchId 幂等合并）
- 收到数据后触发**成本归因引擎**（§4.2）重算该企业的现场浪费地图与画像 V2。

鉴权：请求头 `Authorization: Bearer amk_xxx` → 校验令牌→解析出 `enterpriseId` 与允许的 `source`，防止 A 企业的工具往 B 企业灌数据。

### 9.5 工具侧：连接器 SDK（按各自技术栈各一份薄实现）

同一份逻辑，四种语言/运行时各落一个轻量实现（约 100 行），互不影响主程序：

| 工具 | 运行时 | 连接器落点 | 触发时机 |
|---|---|---|---|
| Worktime | Python/FastAPI | `agent/amiba_connector.py` | 每次 `summarize()` 完成后 |
| APS | Node | `app/amiba-connector.ts` | 每次排产/OEE 计算完成后 |
| BOM | Python/FastAPI | `apps/api/amiba_connector.py` | BOM 定稿/余料登记后 |
| Lean | Node/Express | `packages/core/amiba-connector.ts` | 诊断/报告生成后 |

Python 侧示意（其它栈同构）：

```python
# agent/amiba_connector.py —— 工具对阿米巴零硬依赖，仅靠环境变量激活
import os, json, httpx
def push(metrics: list[dict], waste: list[dict], batch_id: str):
    ep = os.getenv("AMIBA_ENDPOINT")
    if not ep or os.getenv("AMIBA_SYNC_MODE", "off") != "push":
        return  # 独立模式：直接返回，工具行为不变
    payload = {
        "source": os.getenv("AMIBA_SOURCE", "worktime"),
        "enterpriseId": os.getenv("AMIBA_ENTERPRISE_ID"),
        "batchId": batch_id, "schemaVersion": "v2",
        "metrics": metrics, "wasteItems": waste,
    }
    try:
        httpx.post(f"{ep}/api/ingest", json=payload, timeout=5,
                   headers={"Authorization": f"Bearer {os.getenv('AMIBA_TOKEN')}"})
    except Exception as e:
        _queue_for_retry(payload)   # 旁路：失败不影响主流程，落本地重试队列
```

关键点：连接器内部把工具的**原生输出**翻译成 §3.3 的中间模型（adapter），所以工具内部数据结构怎么改都不会破坏阿米巴契约。

### 9.6 能力发现与注册（让阿米巴"知道"接了什么）

工具启动时 `POST /api/connectors/hello`：

```jsonc
{ "source": "worktime", "version": "1.4.0", "enterpriseId": "ent_xxx",
  "capabilities": ["labor_load_rate", "cycle_time", "pmts_compare"] }
```

阿米巴据此在诊断引擎里**动态点亮现场层能力**：接了 Worktime，"人维-人员冗余诊断"才显示"已可实测"，否则提示"需安装视频机位或手工录入"。这让 §2.1 的双轨诊断按实际接入情况自适应——这正是"插拔"在用户界面上的体现。

### 9.7 端到端数据流（以「人员冗余诊断」为例）

```
1. 工厂现场架机位 → Worktime 独立分析视频 → 工时负荷率 61%（独立模式已可用）
2. 该工厂已购阿米巴 → Worktime 配上 AMIBA_* 环境变量
3. Worktime summarize() 后，连接器 push: FactorMetric{man, labor_load_rate, 61%}
4. 阿米巴 /api/ingest 校验令牌 → 落库 data/factory/ent_xxx/
5. 成本归因引擎：负荷率 61% vs 基准 85% → 富余 ~8 人 → WasteItem ≈¥96万/年 → 归"总装阿米巴"
6. 画像 V2 的"人"维自动更新；规划引擎把"消化富余人力"排进改善优先级
7. Lean 生成改善方案 → 落地后复测 → 验证改善金额（闭环）
```

整条链路里，**Worktime 不知道阿米巴怎么算成本，阿米巴不知道 Worktime 怎么切视频**——两边只通过 ingest 契约耦合，这就是插拔式的价值。

### 9.8 落地顺序建议

1. 先在阿米巴侧实现 `POST /api/ingest` + `/api/ingest/file` + 令牌鉴权 + 落库（§9.4）。
2. 用 **L1 文件导入**让任一工具（建议先 Worktime，价值最直观）导出一份样例 JSON 跑通契约。
3. 给该工具加 **L2 连接器 SDK + 环境变量**，实现自动推送。
4. 加 `/api/connectors/hello` 与管理端接入状态页（§9.6）。
5. 逐个复制到 APS / BOM / Lean（连接器同构，主要工作是各自的 adapter 翻译）。

## 十、第五层：阿米巴结果层（杜邦归集树）

V2.1 新增。在四层引擎（诊断→规划→设计→部署）之上加一层**运营结果层**，把所有阿米巴的核算结果
自下而上归集成**杜邦分析树**，并以流程/节点图方式呈现。它是「现场浪费 → 财务结果」贯穿的收口。

### 10.1 核心思想：杜邦树的叶子 = 成本归因引擎的输出

杜邦层不只是展示，它是**聚合引擎**：树叶直接挂到 §4 成本归因引擎产出的成本科目与 `WasteItem`，
于是现场每一项浪费都能沿树向上影响到顶层 ROE，反向也能从一个财务指标下钻到具体的现场浪费与责任巴。

```
            公司顶层巴 · ROE
        ┌─────────┴─────────┐
     净利率          总资产周转率          权益乘数
   ┌───┴───┐         ┌───┴───┐         （仅公司层有意义）
  收入   总成本     收入   总资产
          │
   ┌──────┼──────────────┐
  人工   材料   制造费用 ...（成本科目）
   │      │       │
  WasteItem 叶子（来自成本归因引擎，含 baseline/current/target 三值）
```

### 10.2 分层杜邦：不同巴用不同树（修正经典杜邦）

「权益乘数（财务杠杆）」对内部阿米巴单元没有意义，必须分层：

| 阿米巴类型 | 采用的树 | 顶层指标 | 说明 |
|---|---|---|---|
| 公司顶层巴 | 完整 ROE 杜邦树 | `ROE = 净利率 × 总资产周转率 × 权益乘数` | 唯一保留杠杆的一层 |
| 利润中心巴（营销/制造） | RONA / ROIC 树 | `内部利润率 × 资产周转率` | 去掉杠杆 |
| 成本/支持中心巴（品质/工程/职能） | 服务收费树 | `成本达成率 + 服务单价 × 服务量` | 无收入，按服务计费 |

### 10.3 流程架构呈现

杜邦树本质是一棵计算 DAG，用现有 G6 组件（`G6Network` / `G6Mindmap`）渲染为可下钻的节点图：
点击任一财务节点 → 展开其驱动科目 → 再下钻到现场 `WasteItem` 与责任巴。每个节点同时显示
`baseline / current / target` 三值与达成率，天然支持「实施前后对比」（见 §11.4）。

## 十一、OTD 诊断模式（单阿米巴价值流诊断）

V2.1 新增。诊断引擎在「管理六维」「现场 5M1E」之外新增第三种模式：以**订单→交付（OTD）**价值流
把单个阿米巴的真实工作拉成一条链，逐节点套用现有三性 + 5M1E + 成本归因方法。

### 11.1 关键架构：OTD 是横轴，阿米巴是纵轴

> 一张订单流经多个阿米巴。**OTD = 横向价值流，阿米巴 = 纵向核算单元**；二者交叉的「格子」里放
> 一个**工具 + 一条成本归因规则 + baseline/current/target 三值**。杜邦树（§10）把所有格子向上归集。

跨巴**交接点**必须同时落两样：**内部转移定价** + **责任归属**（质量/延期/超耗算谁的），否则前后对比算不清账。

### 11.2 离散制造 OTD 流程（汽车零部件 / 量产 MTO 版）

> 首发面向汽车零部件量产场景，已**去掉 ETO 设计/出图节点**；量产常为滚动预测 + 看板拉动，
> 故节点 5「采购」对重复件可退化为看板补货。项目制非标（ETO）版后续以可配置分支补充。

| # | 节点 | 核心动作 | 输入交付物 | 输出交付物 | 责任岗位 | 节点 KPI | 三性高风险点 | 可植入工具 |
|---|---|---|---|---|---|---|---|---|
| 0 | 商机/询价 RFQ | 接询价、初判可做性 | 客户图纸/规格 | 询价登记 | 销售 | 询价响应时长 | 完整性：技术要求是否齐 | — |
| 1 | 报价/成本估算 | 估 BOM、估工时、定价（含年降测算） | 图纸、历史成本 | 报价单、估算成本 | 销售+工程 | 报价准确率、报价周期 | 正确性：成本估漏（辅料/工装/年降） | **BOM**、benchmark |
| 2 | 订单评审与接单 | 合同/技术/交期三评审 | 客户 PO/预测、报价 | 评审记录、交期承诺 | 商务+计划 | 接单准确率、超期接单率 | 合理性：盲目承诺交期 | — |
| 3 | 工艺与 BOM 准备 | 工艺路线、制造 BOM、工装、定额 | 设计 BOM/图纸 | 工艺路线、制造 BOM、定额 | 工艺 | BOM 准确率、工艺齐备率 | 完整性：缺工序/缺定额 | **BOM** |
| 4 | 主计划与排产 | MPS、有限产能排产、换线优化 | 订单/看板、产能、BOM | 生产计划、工单 | 计划/PMC | 计划达成率、换线时间(SMED) | 合理性：瓶颈/换线未优化 | **APS** |
| 5 | 物料需求与采购 | MRP/看板补货、跟催 | BOM、库存、计划 | 采购订单、到货计划 | 采购 | 采购准时率、呆滞率 | 正确性：MOQ/提前期错致断料 | APS(库存联动) |
| 6 | 齐套与备料/下料 | 齐套检查、排料下料 | 工单、物料 | 齐套报告、下料件 | 仓储+下料 | 齐套率、套料利用率 | 合理性：未齐套先开工 | **Nesting**、APS、BOM(余料) |
| 7 | 生产制造 | 冲压→焊接→涂装→总装 等多工序 | 齐套物料、工艺 | 在制品、完工品 | 制造各巴 | OEE、单位工时产值、WIP | 正确性：工序参数错 | **Worktime**(工时/负荷)、**APS**(进度/OEE) |
| 8 | 过程与成品质量 | IPQC/FQC、不良处理 | 完工品、检验标准 | 检验记录、不良单 | 品质 | 一次交检合格率、PPM | 完整性：缺检验/缺管控点 | **LeanAI**(8D/根因) |
| 9 | 入库/包装 | 入库、包装、标识 | 合格品 | 入库单、成品 | 仓储 | 库存周转、错发率 | 正确性：错料/错标 | APS(库存) |
| 10 | 发运/物流 | 排车、发运、签收 | 成品、发运计划 | 发货单、签收 | 物流 | 发运准时率 | 合理性：急运成本失控 | APS |
| 11 | 交付确认/开票/回款 | 客户验收、开票、催款 | 签收、合同 | 验收单、发票、回款 | 商务+财务 | OTD率、回款周期(DSO) | 完整性：缺验收致拖款 | 财务接口 |
| 12 | 售后/复盘/知识沉淀 | 客诉处理、复盘 | 全流程数据 | 复盘报告、知识库 | 全员 | 客诉率、复盘录入率 | 合理性：经验不沉淀 | **LeanAI**(知识库/DMAIC) |

**全链贯穿指标**：订单履约周期(OTD L/T)、准时交付率(OTD率)、流动效率(增值时间÷总周期)、订单毛利达成率。

### 11.3 工具植入节点（用户按需启用）

每个 OTD 节点声明一组「可植入工具」（上表末列）。复用 §9.6 的**连接器能力发现**：
- 工具已接入且上报过能力 → 该节点工具**可启用**（点亮）
- 未接入 → 显示「可启用，需先接入 XXX」（置灰 + 引导到「工具接入」页）

用户进入某个阿米巴的 OTD 视图后，可逐节点勾选启用工具，实现「按需选择工具进行管控」。

### 11.4 实施前后对比（基线冻结 + 归因）

- **基线冻结**：诊断启动时对每个节点 KPI 与成本科目快照存 `baseline`；改善后测 `current`；目标存 `target`。
- **干预归因**：记录每次干预（启用了哪个工具/做了哪条改善/时间点），把 `current-baseline` 的改善额
  关联到具体干预，避免把自然波动算成工具功劳（反事实问题）。
- **粒度对齐**：OTD 多为按单口径（L/T、齐套率），杜邦为周期口径（周转、OEE）；中间加一层
  「按单 → 周期」汇总规则，使两层对得上。
- **输出**：节点级、阿米巴级、公司级三档「实施前 vs 后」对比，并沿杜邦树展示改善对 ROE 的最终贡献。

### 11.5 数据模型增量（V2.1）

- `otd-types.ts`：`OtdNode`（含 io 交付物、责任岗位、KPI、三性风险、可植入工具）、`OtdInstance`（按单）、`OtdTemplate`（行业模板）。
- `metric` 增加 `baseline / current / target` 三值与 `interventions[]`。
- `dupont.ts`：分层杜邦树定义 + 由 `WasteItem`/成本科目聚合的计算函数。
- 页面：阿米巴 OTD 价值流视图（节点链 + 工具开关 + 前后对比）、结果层杜邦看板。

## 十二、信息架构与数据模型重构（V2.2）

V2.2 把「四个并列引擎 + 散点 5M1E」重构为**一条有因果顺序的价值链**，OTD 成为贯穿脊柱，
5M1E 降为节点标签，工具成为节点上的可选执行器。本章是接口级重构的对齐稿。

### 12.1 新信息架构（两层导航 + 主线）

**顶层（咨询顾问视角）**：轻量企业台账——选择/管理多家客户企业。
**企业内（单企业工作区）**：以诊断为首页，沿主线推进：

```
[顶层] 企业台账（多客户）
   └─进入某企业──► [企业工作区]
        首页 = 诊断  →  企业画像  →  规划引擎          →  设计引擎          →  部署引擎
        多轮探测       就绪度+      OTD价值流(取代业务流程)  阿米巴切割 + 与OTD/    按 方案×规则×工具
        现场扫描       浪费地图     + 数据采集成熟度档次     数据采集统筹总体设计   生成实施方案
                                          └──── OTD 价值流贯穿四引擎（同一对象，不同状态）────┘
                                                       ▼
                                          结果层杜邦树：全链结果归集到 ROE
```

导航调整：原「总览」拆为**顶层台账**（多企业）+ **企业首页=诊断**；
「OTD 价值流」「结果层」「工具接入」不再是平级菜单，而是嵌入对应引擎/工作区。

### 12.2 OTD = 跨引擎贯穿件（字段归属与状态机）

同一条 OTD 在四个引擎里是**同一对象的不同状态**，各引擎只读写自己负责的字段：

| 引擎 | 对 OTD 的读写 | 负责字段 |
|---|---|---|
| 诊断 | 写 as-is | `kpi.baseline / kpi.current`、`riskProp`、三性诊断结论 |
| 规划 | 写 to-be | `kpi.target`、节点保留/合并、`dataLevel`（数据采集档次 L1/L2/L3） |
| 设计 | 叠加归属 | 节点 `amibaId`、交接点 `transferPrice`、`responsibleAmibaId` |
| 部署 | 读全部 | 由 `tools[].enabled` + 规则生成 `implementationPlan` |

> 原则：OTD 是中心对象（`OtdTemplate/OtdNode`），不归任何单一引擎所有；引擎是它的「视图 + 编辑器」。
> 这解决了「设计引擎要和业务流程统筹」时数据不一致的问题。

OTD 节点状态机：`draft（库模板）→ diagnosed（诊断填 as-is）→ planned（规划填 to-be）→ designed（设计填归属）→ deploying（部署生成方案）`。

### 12.3 5M1E 归并：节点级统一指标模型（数据接口合并）

**决策：5M1E 降为 OTD 节点上的标签，指标统一挂节点。** 这要求把现有两套并行数据合并为一套：

| 现状（V2.1，两套） | 重构后（V2.2，一套） |
|---|---|
| `FactorMetric`（按企业 + 5M1E 要素，工具 ingest 落点） | 并入节点 KPI：`NodeKPI` 增加 `factor`、`source`、`capturedAt` |
| `WasteItem`（按企业散点） | 改为挂在节点上：`OtdNode.wasteItems[]` |
| `NodeKPI`（按节点，OTD 页编辑） | 成为唯一指标载体，含三值 + 来源 + 要素标签 |

**ingest 落点改变（核心接口变更）**：工具上报不再落「企业级散点」，而是**回填到「对应节点的对应 KPI 的 current 值」**：

```jsonc
POST /api/ingest        // 信封增加 nodeKey/kpiKey 定位
{
  "source": "worktime",
  "enterpriseId": "ent_x",
  "templateId": "otd_x",          // 落到哪条 OTD
  "updates": [
    { "nodeKey": "production", "kpiKey": "oee", "value": 61, "capturedAt": "..." },
    { "nodeKey": "production", "kpiKey": "labor_value", "value": 138 }
  ]
}
```

这样「工具自动采集 → 节点 KPI.current 回填 → OTD 前后对比 → 杜邦叶子聚合 → ROE」一条线真正闭环。
`factory-types.ts` 的 `FactorMetric/WasteItem` 保留为「中间模型/对外契约」，但内部落库改为节点级。

### 12.4 部署规则引擎（决策表 + 人工退化 + 导入导出）

部署引擎按 `节点 × 工具是否启用且已接入 → 实施方式` 生成实施方案：

| 节点 | 选了对应工具（自动） | 没选工具（人工退化） |
|---|---|---|
| 生产制造·工时 | 接 Worktime 自动采工时/OEE | 生成《工时人工采集模版》+ 导入接口 |
| 齐套/下料·利用率 | 接 Nesting/APS 自动算 | 生成《套料利用率台账模版》+ 导入 |
| 质量·根因 | 接 LeanAI 自动 8D | 生成《不良/8D 手工模版》+ 导入 |
| 采购·呆滞 | 接 APS 库存联动 | 生成《库存/呆滞台账模版》+ 导入 |

规则引擎结构：`rules[] = { match: {nodeKey, factor, toolAvailable}, plan: {method, templateRef, importApi} }`。
极端情况「客户一个工具都不选」→ 全链走人工采集模版，系统生成整套 Excel 模版包 + 提供
`POST /api/ingest/file`（设计文档 §9.4 规划过，**本次需实现**）做导入，与工具自动采集殊途同归。

输出物：`ImplementationPlan`（按节点的实施动作、采集方式、模版/接口、责任人、依赖、配套工具）。

### 12.5 接口与代码改动清单（迁移）

| 区域 | 改动 |
|---|---|
| `otd-types.ts` | `NodeKPI` 加 `factor/source/capturedAt`；`OtdNode` 加 `wasteItems[]/dataLevel/amibaId`；交接点 `transferPrice` |
| `factory.ts` | ingest 落库从「企业散点」改为「节点 KPI 回填」；保留对外契约 |
| `/api/ingest` | 信封支持 `templateId + updates[{nodeKey,kpiKey,value}]` |
| `/api/ingest/file`（新增） | 人工模版导入，复用同一回填逻辑 |
| `dupont.ts` | 叶子 `current` 由节点 KPI/wasteItems 聚合得出（替换演示财务） |
| 部署引擎 | 新增 `rules.ts`（决策表）+ `ImplementationPlan` 生成 + 模版导出 |
| 导航/页面 | 总览拆两层；OTD/结果层/工具接入嵌入对应引擎；诊断成为企业首页 |
| 现有 `(app)/page→dashboard` | 改为顶层企业台账；企业工作区路由如 `/e/[id]/diagnosis` 等 |

### 12.6 分阶段落地建议

1. **P1 数据模型合并**：`NodeKPI` 吸收 factor/source；ingest 改为节点回填；`/api/ingest/file`。（地基）

3. **P3 OTD 贯穿件**：四引擎按 §12.2 字段归属各自读写 OTD；状态机。
4. **P4 部署规则引擎**：决策表 + 人工模版包导出 + 实施方案生成。
5. **P5 杜邦接线**：叶子由节点真实数据聚合，全链闭环。

## 十三、分层流程 + 自底向上成本汇总 + 实时跟踪（V2.3）

V2.3 把单层 OTD 升级为**可下钻的分层流程**：每个节点可挂子流程（BPMN 泳道图），叶子节点做
数据采集与成本归因，成本自底向上层层汇总到本页 OTD 流程图并**准实时跟踪**。

### 13.1 分层流程模型（递归子流程）

```
OTD 顶层节点 (13)
   └─ 子流程 Subflow { lanes[](部门), nodes[], edges[] }
         └─ 节点可再挂子流程 …（建议最多 3 层，防汇总/渲染爆炸）
               └─ 叶子节点 = 成本最小单元（数据采集 + 成本归因）
```

- `ProcessNode` 递归：节点可选挂一个 `Subflow`；无子流程者即**叶子**。
- 每层节点都存 `计划 / 实际 / 目标` 三值（沿用 V2.2 基线冻结）。

### 13.2 BPMN 泳道 = 部门；转让价分层

- 子流程用 **BPMN 泳道图**，每条泳道 = 一个**部门**；活动放在其部门泳道内，按顺序流连接。
- 维护**部门目录** + **部门→阿米巴映射**（部门归属于某阿米巴）。
- **转让价分层（已定）**：
  - **跨阿米巴**的交接 → **正式内部转让价**（重，进核算）。
  - **同阿米巴内跨部门**的交接 → **轻量协作工时计价**（可开关，趋势用，不强制进核算）。
- 顶层 OTD 的阿米巴交接（§11、已实现）与子流程的部门交接，统一由"交接点引擎"按上述规则判定。

### 13.3 文档解析 + 手工绘制（同一模型）

两条入口，产出同一个 `Subflow`：
- **文档解析（首批含 LLM）**：
  - 结构化：`.bpmn` XML、Excel/CSV 流程清单 → 直接高保真映射。
  - 非结构化：**Word/PDF/文本 → LLM 抽取**为 `{lanes, nodes, edges}` 模型。
  - 图片/扫描件（OCR）低保真，后置。
  - **强制人工校正**：解析结果进入可编辑画布确认后才生效（不承诺"任意文档→精确流程"）。
- **手工绘制**：X6 图编辑器（加部门泳道、加活动节点、连顺序流、属性面板）。

### 13.4 叶子节点：数据采集方案 + 成本归因

每个叶子挂一组**成本科目项**（按 5M1E）：

```ts
interface CostItem {
  factor: Factor;                 // 人/机/料/法/环/测
  account: string;                // 成本科目，如 "直接人工"
  capture: "tool" | "manual";     // 采集方式（接 P4 规则引擎）
  source?: ConnectorSource;       // 工具来源 / manual
  rateKey: string;                // 关联费率表的单价 key
  planQty?: number; actualQty?: number;  // 计划/实际用量（工时/面积/度…）
  // 成本 = qty × rate
}
```

- **采集方式**来自部署规则引擎（选了工具→自动；没选→人工模版 + `/api/ingest/file`）。
- **实际值**来自 ingest 回填——**回填定位下沉到叶子**：`{ leafNodeId, costItemKey, actualQty }`。
- **费率表（新增，关键）**：工时单价、设备工时单价、材料单价、能耗单价…；没有费率表则实时成本算不出。
- **计划成本**来源：定额 / 标准工时（BOM、PMTS）→ 作为 baseline。

### 13.5 成本自底向上汇总（Rollup）

```
叶子成本 = Σ CostItem(qty×rate)
   ↑Σ
子流程节点成本 = Σ 子节点
   ↑Σ
OTD 顶层节点成本 = Σ 子流程
   ↑
杜邦树成本科目（§10 叶子）← 按 5M1E/科目归集
```

每层节点输出 `计划 / 实际 / 目标` 与偏差；rollup 引擎在叶子数据变更时自底向上重算并缓存。

### 13.6 准实时成本跟踪（已定：准实时）

- **触发**：工具 ingest 或人工导入落库 → 重算该叶子成本 → 向上 rollup → 写入缓存。
- **前端**：进入页面即拉 + 轮询（非 SSE）。
- **本页 OTD 流程图**：每节点显示`实际 / 计划 + 偏差色`（超支红/结余绿）；顶部**总成本看板**按
  阿米巴 / 部门 / 5M1E 维度切分实际 vs 计划；交接点显示累计转让价金额。

### 13.7 数据模型与接口增量

| 区域 | 改动 |
|---|---|
| `process-types.ts`（新增） | `ProcessNode`(递归) / `Subflow` / `Lane`(部门) / `CostItem` |
| `departments.ts`（新增） | 部门目录 + 部门→阿米巴映射 |
| `rates.ts`（新增） | 费率表（工时/设备/材料/能耗单价） |
| `otd-types.ts` | OtdNode 增加 `subflowId?`；KPI/成本下沉到叶子 |
| `rollup.ts`（新增） | 自底向上成本汇总 + 缓存 |
| `/api/ingest` | 回填定位下沉到 `{leafNodeId, costItemKey, actualQty}` |
| `/api/subflow`（新增） | 子流程 CRUD |
| `/api/parse-doc`（新增） | 文档上传 → 解析（结构化映射 + LLM 抽取）→ 返回 Subflow 草稿 |
| `/api/cost`（新增） | 拉取某企业/某节点的实时 rollup 成本 |
| 页面 | OTD 节点下钻进子流程泳道图；子流程编辑器；叶子成本编辑；总成本看板 |

### 13.8 分阶段落地

1. **P-a** 分层流程模型 + 子流程 CRUD（先手工建：泳道+节点+边）。
2. **P-b** 子流程泳道图渲染 + X6 编辑器（部门泳道、加节点连线、属性面板）。
3. **P-c** 费率表 + 叶子成本科目 + 成本归因 + rollup 引擎（计划/实际/目标）。
4. **P-d** 准实时成本跟踪：ingest 下沉到叶子 → 重算 rollup → 流程图节点偏差色 + 总成本看板。
5. **P-e** 文档解析：先结构化(BPMN/Excel)直映射，再 LLM 解析 Word/PDF + 人工校正画布。

## 十四、实施流水线重构 + 门控 + 产品化数据飞轮（V2.4）

V2.4 把"四个并列引擎"重构为一条**因果闭环的实施流水线**，并以**单个产品**为载体驱动阿米巴落地、沉淀数据。

### 14.1 新实施主线与信息架构

```
（运行驾驶舱·始终可见）────────────────────────────────► 总览
        ▲ 实时成本归集
设计引擎 ──► 规则引擎 ──► 诊断引擎 ──► 企业画像 ──► 部署引擎 ──► 运行
OTD骨架     子流程+成本    成本问题      人机料法环      方案+甘特
+阿米巴/部门  +转让价/费率   +改进方向     +成本六维结论    +落到节点/子流程
```

- **菜单 = 向导式 stepper**：按流程顺序显示「设计→规则→诊断→画像→部署」，每项带**步骤号 + 锁/对勾**；**总览单独置顶作为驾驶舱，始终可点**（不锁）。
- 顺序**前进解锁**，但**后退随时可改**（阿米巴是迭代的）；已完成/当前引擎始终可读，只锁"未达前置条件的引擎进入/编辑"。

### 14.2 门控完成判定（gate）

| 引擎 | 完成判定（达成才解锁下一个） |
|---|---|
| 设计 | OTD 骨架节点已定 + 阿米巴/部门目录已确认 |
| 规则 | 节点子流程/成本要素录入覆盖率 ≥ 阈值（默认叶子 80% 有成本）+ 转让价/费率已设 |
| 诊断 | 已生成诊断结论（成本问题清单 + 改进方向） |
| 画像 | 人机料法环+成本 六维已计算并出结论 |

| 总览 | 始终可见（不设门控） |

### 14.3 各引擎重定义（已按决策固化）

- **设计引擎**＝**阿米巴单元体构建**：根据用户输入的**部门与人员**，设计阿米巴单元、**部门转让价、部门内部成本核算规则**（沿用既有 `/design` 设计引擎功能；**不含 OTD 流程**）。
- **规则引擎**＝**OTD 流程图** + 在其上建**子流程泳道(=部门) + 节点人机料费成本要素 + 部门转让价/费率**，并展示规则与成本状态统计表与覆盖率。（OTD 流程引擎归集于此）
- **诊断引擎**＝规则建成后，基于 **rollup + 待完成清单 + 三性 + benchmark**，输出**所有节点的人/机/料/费成本问题 + 改进方向**。（重建：从"问卷"改为"对已建模型的成本诊断"）
- **企业画像**＝由诊断结果综合成**六维（对齐现场数据）**：人(人效)/机(OEE)/料(利用率)/法(流动效率)/环(能耗)/成本(健康度)，叠加少量组织/文化定性，给出结论。
- **部署引擎**＝综合前三者结论 → 实施步骤/方案/**周期** → 落到 OTD 各节点与子流程 → 形成**甘特图**管理。
- **总览**＝按甘特进度做**实时成本归集统计**（杜邦+rollup 闭环已就绪），展示"改善前→当前→目标"随时间推进曲线，看阿米巴效果。

> 前置补充：在设计引擎前加一个**轻量"立项/可行性"前置**（行业/规模/信息化/变革意愿），避免在错误前提上建模。

### 14.4 甘特—成本联动（关键，避免两张皮）

改善任务必须**可回写成本**：

```ts
interface ImprovementTask {
  id: string; title: string;
  targetNodeId: string;          // 影响哪个 OTD/子流程节点
  targetKpiKey?: string;         // 影响哪个 KPI / 成本科目
  expectedDelta?: number;        // 预期成本变化（元，负=节约）
  owner: string; start: string; end: string; deps: string[];
  status: "todo" | "doing" | "done";
}
```

任务 `done` → 触发该节点成本复测/应用 `expectedDelta` → **rollup 重算** → **总览成本曲线更新**。这样甘特进度与实时成本是同一份数据。

### 14.5 产品化数据飞轮 + 基准库

以**单个产品（产品/产品线的 OTD）= 一个阿米巴项目**为单元；每做完一个，沉淀**可复用基准**：

```ts
interface ProductBenchmark {
  productId: string; industry: Industry; nodeKey: string;
  stdLaborCost: number; stdHours: number; materialYield: number;
  transferPrice?: number; improveBeforeAfter?: { before: number; after: number };
}
```

- **匿名化累积** → 行业 benchmark（越用越准，数据壁垒）。
- **反哺前端**：下一个产品的报价/工艺/BOM/成本估算**复用上一个产品的节点基准** → 接 BOM/APS/Worktime 工具。
- **行业押注**：产品驱动 + 数据复用对**汽车零部件(量产/重复)最有效**；项目制非标(ETO)复用度低，沉淀到部门/工序级即可。
